# 代码审查

## 使用场景
用户希望你审查一个或多个代码文件，找出潜在问题并给出改进建议。

## 步骤

1. 确定要审查的文件路径。如果用户没有指定具体路径，先用 `file_list` 了解项目结构。

2. 使用 `file_read` 读取目标文件内容：
   ```
   file_read(path="<文件路径>")
   ```

3. 从以下维度审查代码：

   **正确性**
   - 逻辑错误、边界条件、空值处理
   - 类型错误、参数校验

   **安全性**
   - SQL 注入、XSS、路径遍历
   - 硬编码密钥、敏感信息泄露
   - 输入验证不足

   **可维护性**
   - 命名规范、代码重复
   - 函数过长、职责不清
   - 缺少必要的错误处理

   **性能**
   - 明显的性能问题（N+1 查询、不必要的循环等）
   - 资源泄露风险

4. 输出结构化的审查报告：
   - 🔴 **严重问题**：必须修复的 bug 或安全漏洞
   - 🟡 **建议改进**：代码质量和可维护性建议
   - 🟢 **做得好的地方**：值得肯定的实践

5. 对每个问题给出具体的修改建议或示例代码。

## 注意事项
- 聚焦于有实际影响的问题，避免吹毛求疵
- 给出具体的改进代码，而不只是抽象建议
- 如果文件较多，按重要性优先审查核心逻辑文件
